# Бенчмарк токенов Archcore — scale-выводы

[English](FINDINGS_SCALE.md) · **Русский**

> Сырые выводы scale-стенда: кривая перелома, реалистичный ворклоад и механика.
> Краткий обзор: [`../SUMMARY.ru.md`](../SUMMARY.ru.md) · полный анализ: [`../ANALYSIS.ru.md`](../ANALYSIS.ru.md).

**TL;DR.** На задаче с фиксированным качеством **стоимость Archcore за задачу не зависит от
размера базы, а preload в `CLAUDE.md` растёт линейно** — поэтому Archcore дешевле preload при
любой базе больше **~27 доков**, и в **4.6 раза** при 320 доках. Против «markdown-файлы + греп»
Archcore находится на **примерном паритете** по всему тестовому диапазону.

Стенд: chi @ `3b171578`, модель=sonnet. KB = 320 синтетических мультидоменных доков (~550 токенов
каждый, реалистичный размер ADR/правила). Точные токены из `claude -p --output-format json`.
5 arm'ов, 5 trial/ячейку (кроссовер), рандомизированный порядок. Контроль качества: каждая задача
— произвольный токен, спрятанный в одном документе; оценка по точному совпадению.

## Arm'ы

| Arm | Доступ к знаниям | Моделирует |
|-----|-----------------|------------|
| A — холодный | нет | репо без агентных доков (пол) |
| B1 — preload | все N доков в `CLAUDE.md` | «всё в контекст» |
| B2 — файлы + индекс | N доков как файлы + идеальный `CLAUDE.md`-индекс | доки-файлы, курируемые |
| B3 — blind grep | N доков как файлы, без индекса | доки-файлы, агент ищет Grep'ом |
| C — archcore | N доков как файлы + Archcore MCP | Archcore |

## Метрика стоимости

Используем **realistic-метрику**: свежая сессия на каждую задачу (холодно между задачами),
но ходы внутри задачи — кеш-хит (они идут в одном CLI-вызове). Восстанавливается независимо
от порядка из `context_tokens/turns` и числа ходов.

Оба наивных предела приведены в raw-анализе для прозрачности:
- *cold* завышает стоимость многоходовых arm'ов.
- *warm* (реально выставленный счёт) занижен из-за кросс-рунового кеш-перетекания.

## Кроссовер — фиксированная задача, рост KB (realistic за задачу, $)

| N (доков) | A холод | B1 preload | B2 индекс+греп | B3 blind grep | **C archcore** |
|---|---|---|---|---|---|
| 1   | 0.204 ✗ | **0.102** | 0.111 | 0.103 | 0.127 |
| 20  | 0.183 ✗ | 0.134 | **0.113** | 0.121 | 0.127 |
| 80  | 0.188 ✗ | 0.238 | **0.119** | 0.123 | 0.127 |
| 160 | 0.195 ✗ | 0.386 | 0.130 | **0.124** | 0.127 |
| 320 | 0.228 ✗ | 0.681 | 0.152 | **0.112** | 0.127 |

✗ = задача провалена (холодный arm не имеет знаний; pass 0–20%).

**Точки перелома — N, при котором C (archcore) становится дешевле или равен:**

| vs бейзлайн | realistic N* |
|---|---|
| C vs B1 preload | **≈ 27 доков** |
| C vs B2 индекс+греп | ≈ 272 дока |
| C vs B3 blind grep | примерный паритет по всему диапазону |

```
realistic за задачу vs N
B1 preload   N=1 ▏$0.10   N=80 ▍$0.24   N=320 █$0.68     (линейно в N — растёт без предела)
C archcore   N=1 ▎$0.13   N=80 ▎$0.13   N=320 ▎$0.13     (плоско — не зависит от N)
B2 индекс    N=1 ▎$0.11   N=80 ▎$0.12   N=320 ▎$0.15     (почти плоско; индекс медленно растёт)
B3 греп      N=1 ▎$0.10   N=80 ▎$0.12   N=320 ▎$0.11     (плоско)
```

## Воркнагрузка — N=80, полная сюита из 20 задач (B2/B3/C × 3 trial)

| Arm | сюита realistic $ | (cold) | (warm) | pass | мед. ходов/задача |
|---|---|---|---|---|---|
| B3 blind grep | **2.294** | 5.040 | 0.849 | 100% | 3 |
| B2 индекс+греп | 2.375 | 3.495 | 0.475 | 100% | 2 |
| C archcore | 2.716 | 7.370 | 1.129 | 100% | 5 |

Все три дают верный ответ на каждую задачу (одинаковое качество). По raw realistic-стоимости
**B3 и B2 дешевле C на ~15%** — встроенный Grep находит нужный документ за 2–3 хода, Archcore
делает ~5 ходов. Смотри §Механизм ниже.

## Механизм: overhead ToolSearch

5-ходовой цикл Arm C в воркнагрузке: `ToolSearch(search_documents)` → `search_documents`
→ `ToolSearch(get_document)` → `get_document` → ответ. Два из этих ходов — ленивая загрузка
схем MCP-тулов (`ToolSearch`) в Claude Code — overhead, который встроенный Grep не платит.

С `search_documents(mode=full)` (тело документа прямо в результате) `get_document` не нужен
→ 3 хода, паритет с грепом. Таблица воркнагрузки выше отражает 5-ходовое поведение v1;
таблица кроссовера использует 3-ходовые / $0.127 числа (mode=full, все N).

## Вывод — три находки

1. **Стоимость Archcore плоская; preload — линейный.** Archcore обгоняет `CLAUDE.md` preload
   примерно с **27 доков** и дешевле в **4.6 раза при 320 доках** ($0.127 против $0.681).
2. **Archcore на паритете с «файлы + греп».** Оба — $0.11–0.13/задачу по всему тестовому
   диапазону (1–320 доков), без необходимости поддерживать индекс.
3. **Знания должны где-то жить.** Холодный arm — одновременно самый дорогой и проваливший
   задачу.

## Что можно / нельзя заявлять

- ✅ **МОЖНО:** «Archcore держит стоимость плоской с ростом базы; `CLAUDE.md` preload растёт
  линейно — при базе больше ~27 доков Archcore дешевле, в 4.6 раза при 320 доках».
- ✅ **МОЖНО:** «Archcore на паритете с "файлы + греп" и не требует обслуживания индекса».
- ✅ **МОЖНО:** «Archcore стабилен (~3–5 ходов) на разнородных задачах при одинаковом качестве».
- ❌ **НЕЛЬЗЯ:** «Archcore экономит токены против любого бейзлайна» — не выигрывает у грепа.
- Честный результат — **кривая перелома**, а не одно значение «−X%».

## Оговорки / ограничения

1. Одна модель (sonnet), одно репо (chi). Только однофакторные задачи.
2. Индекс B2 считается идеально поддерживаемым и бесплатным; реальная стоимость поддержки
   здесь не учтена.
3. Realistic-метрика — реконструкция (предполагает ~постоянный prefix на ход);
   cold/warm-границы: `results/FINDINGS_SCALE_raw.md`.
