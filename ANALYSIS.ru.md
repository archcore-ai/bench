# Бенчмарк токенов Archcore — Полный анализ

[English](ANALYSIS.md) · **Русский**

> **Цель:** воспроизводимо установить, экономит ли Archcore токены на реальной кодовой базе
> при фиксированном качестве — и если да, то где именно. Документ содержит все цифры и честные
> выводы, включая то, что заявлять нельзя.

Модель: claude-sonnet-4-6. Токены — точные, из `claude -p --output-format json` (`.usage`).

## 2. CLI-бенчмарк

### 2.1 Arm'ы

Все arm'ы используют один источник (`go-chi/chi`) и одни инструменты (`Read/Grep/Glob`).
B1/B2/B3/C несут **одинаковые документы** — различается только способ доступа. Arm A — пол.

| Arm | Знания | Индекс | Доп. инструменты | Моделирует |
|-----|--------|--------|-----------------|-----------|
| **A — холодный** | нет | — | — | репо без агентных доков |
| **B1 — preload** | все N доков в `CLAUDE.md` | n/a | — | «свалить всё в контекст» |
| **B2 — файлы + индекс** | N доков как файлы + индекс в `CLAUDE.md` | да | — | доки-файлы с картой |
| **B3 — файлы + греп** | те же N доков как файлы, без индекса | нет | — | доки-файлы, поиск Grep'ом |
| **C — archcore** | те же N доков как файлы | нет (авто) | Archcore MCP | поиск Archcore |

B2 и B3 включены оба, потому что конкурент Archcore — не только наивный preload, но и «просто
держать markdown в репозитории». B2 даёт подходу с файлами+греп идеальный свободный индекс
(самый жёсткий бейзлайн). B3 убирает индекс — агент находит документы через встроенный Grep.
C-vs-B3 изолирует вопрос: «побеждает ли MCP-поиск Archcore обычный Grep над теми же файлами?»

### 2.2 База знаний и контроль качества

320 синтетических документов по 5 доменам (middleware, routing, errors, logging, testing),
**~550 токенов каждый** (реалистичный ADR/правило: Context / Decision / Consequences /
Alternatives / Example / References). Каждый документ кодирует одно нормативное правило,
ответ на которое — **произвольный токен в секции Decision** — в заголовке и excerpt'е его нет,
поэтому только результата поиска недостаточно. Агент обязан открыть документ.

Градуировка по точному вхождению токена: верный ответ = знание реально получено, а не угадано.

### 2.3 Метрика стоимости

Три метрики, все восстановлены независимо от порядка запусков:

- **realistic (заголовочная):** свежая сессия на каждую задачу (кеш холодный между задачами —
  реальность многосессионного режима), но ходы внутри задачи кешируются (они идут в одном CLI-вызове).
  ```
  ctx_in    = input_tokens + cache_creation + cache_read
  prefix    = ctx_in / num_turns
  realistic = prefix × (1.25 + 0.10 × (turns − 1)) × $3/M + output × $15/M
  ```
- **cold:** `ctx_in × $3/M + output × $15/M` — верхняя граница; завышает стоимость многоходовых arm'ов.
- **warm:** `total_cost_usd` — нижняя граница; занижена из-за кросс-рунового кеш-перетекания.

Все таблицы ниже — realistic-метрика. Границы cold/warm: `scale/results/FINDINGS_SCALE_raw.md`.

### 2.4 Результаты кроссовера

| 1         | 0.204 ✗ | **0.102**  | 0.111          | 0.103         | 0.127          |
| 20        | 0.183 ✗ | 0.134      | **0.113**       | 0.121         | 0.127          |
| 80        | 0.188 ✗ | 0.238      | **0.119**       | 0.123         | 0.127          |
| 160       | 0.195 ✗ | 0.386      | 0.130          | **0.124**     | 0.127          |
| 320       | 0.228 ✗ | 0.681      | 0.152          | **0.112**     | 0.127          |

### 2.5 Результаты воркнагрузки

Фиксированный N=80; 20 задач по всем 5 доменам; arm'ы B2/B3/C; 3 trial.

| Arm | Сумма realistic $ | Pass rate | Медиана ходов/задача |
|-----|-------------------|-----------|---------------------|
| B3 — blind grep | **$2.294** | 100% | 3 |
| B2 — индекс+греп | $2.375 | 100% | 2 |
| C — archcore | $2.716 | 100% | 5 |

Все три дают верный ответ на каждую задачу. B3 и B2 дешевле C на ~15%.

### 2.6 Механизм: количество ходов и налог ToolSearch

| Arm | Ходы | Контекст (медиана, N=80) | Растёт с N |
|-----|------|--------------------------|------------|
| A холод | ~13 | ~352 000 | да |
| B1 preload | **1** | 27K → 182K | **линейно** (несёт весь KB) |
| B2 индекс+греп | 2 | 54K–74K | почти плоско |
| B3 blind grep | 3 | 80K–84K | плоско |
| C archcore | **3** | **~93K** | **плоско** |

Arm C делает 3 хода: `ToolSearch(search_documents)` → `search_documents(mode=full)` → ответ.
`search_documents` с `mode=full` возвращает тело документа прямо в результате, поэтому
отдельный вызов `get_document` не нужен. Это совпадает с 3-ходовой схемой B3.

Без `mode=full` Arm C делал бы 5 ходов (дополнительные `ToolSearch(get_document)` +
`get_document`): Claude Code загружает схемы MCP-тулов лениво — перед каждым вызовом тула
идёт ход загрузки схемы. `mode=full` сворачивает поиск в один MCP-вызов, убирая этот overhead.

Постоянный вклад Archcore MCP (схемы, инструкции сервера) — ~638 токенов/ход дополнительно к
бейзлайну, но это константа, которая сокращается при сравнении C с собой при разных N.

## 3. Plugin-бенчмарк

### 3.1 Методология

Plugin добавляет `/archcore:context` в Claude Code через `--plugin-dir`. В headless-сессии
`claude -p` скилл вызывается первой строкой промпта:

```
/archcore:context middleware

Для каждого вопроса ответь только точным токен-значением:
1. [вопрос]
...
```

Два arm'а:
- **D-arm (Plugin):** `--plugin-dir /path/to/plugin` + префикс `/archcore:context <домен>`
- **C-arm (контроль батча):** те же N вопросов, без plugin, один вызов `claude -p`

Бейзлайн: **C-separate** = один `claude -p` на вопрос (CLI, $0.127/вопрос).

**Дизайн:** 3 trial × 5 доменов × 4 размера батча (N ∈ {1, 2, 4, 8}) × 2 arm = **120 измерений**.
Pass rate: 100% по всем условиям.

### 3.2 Механизм сессионной амортизации

Фиксированный overhead на сессию — схемы MCP-тулов, инструкции скилла, system prompt,
контекст исходников — составляет ~88–190k токенов в зависимости от arm'а. Этот overhead
оплачивается один раз (cache_creation на ходу 1) и кешируется на всех последующих ходах
сессии (cache_read по ставке 0.10×). При N вопросах в одной сессии:

```
total_cost(N) ≈ overhead_cost + N × marginal_cost_per_answer
per_question(N) = total_cost(N) / N
```

Маржинальная стоимость каждого дополнительного вопроса — ~$0.002–0.005 (только output-токены).
С ростом N стоимость за вопрос стремится к этой маржинальной стоимости.

**Количество ходов D-arm:** скилл `/archcore:context` вызывает `search_documents` без `mode=full`,
возвращая top-5 excerpt'ов. Токены-ответы лежат в секции Decision и в excerpt'е не появляются,
поэтому модель делает `get_document` на каждый вопрос (~2 хода каждый). При N=4: 3 хода
загрузки контекста + 8 ходов на 4 вопроса + 1 ход ответа = ~12 ходов.

**Количество ходов C-arm:** модель вызывает `search_documents(mode=full)`, получает тела всех
подходящих документов и отвечает на все N вопросов в одном ходу → 5 ходов при любом N.

Почему D-arm выигрывает у C-arm при N≥4 в realistic-метрике: при 12 ходах prefix D-arm'а
на ход — ~14k токенов (11 ходов по ставке 0.10× cache-read). При 5 ходах prefix C-arm'а —
~18k токенов (4 хода по cache-read ставке). Кеш-модель поощряет больше ходов; при N≥4 это
перевешивает бо́льший абсолютный контекст D-arm'а.

### 3.3 Стоимость за вопрос по размеру батча

**Сводная таблица — медиана по 3 trial, всем 5 доменам:**

| Вопросов/сессия | D-arm $/q | D IQR | C-arm $/q | C IQR | D/C | экономия vs C-sep |
|-----------------|-----------|-------|-----------|-------|-----|-------------------|
| 1 | $0.163 | ±0.034 | $0.150 | ±0.001 | 1.09 | D: −29%, C: −18% |
| 2 | $0.075 | ±0.007 | $0.067 | ±0.014 | 1.13 | D: −41%, C: −47% |
| 4 | $0.032 | ±0.001 | $0.033 | ±0.008 | 0.97 | D: **−74%**, C: **−74%** |
| 8 | $0.015 | ±0.000 | $0.014 | ±0.009 | 1.05 | D: −88%, C: −89% |

При N=1 оба arm'а дороже C-separate ($0.127): C-separate использует оптимизированную
3-ходовую сессию на один вопрос, а батч N=1 несёт дополнительный overhead.

**Эквивалент воркнагрузки — 20 задач (5 доменов × 4 вопроса):**

| Стратегия | Итого | vs C-sep 20×1 |
|-----------|-------|---------------|
| C-separate 20×1 | $2.540 | бейзлайн |
| D-батч 5×4 | **$0.649** | **−74%** |
| C-батч 5×4 | $0.722 | −72% |

D-батч дешевле C-батча на 10% на полной воркнагрузке.

### 3.4 D vs C по доменам при батче 4

| Домен | D-arm $/q | C-arm $/q | Победитель | Δ |
|-------|-----------|-----------|------------|---|
| middleware | $0.033 | $0.033 | D | 1% |
| routing | $0.033 | $0.033 | D | 1% |
| errors | $0.032 | $0.041 | **D** | **20%** |
| logging | $0.032 | $0.033 | D | 3% |
| testing | $0.032 | $0.041 | **D** | **21%** |

Plugin выигрывает по всем 5 доменам. Для `errors` и `testing` (ADR-документы со сложными
формулировками тем) D-arm дешевле на 20–21% — скилл контекста помогает агенту точнее
находить нужные документы без лишних итераций поиска.

**Стабильность:** IQR у D-arm = $0.001/q при N=4; у C-arm = $0.008/q — Plugin в 8× стабильнее.

(−88% против CLI-отдельных). Точка безубыточности — ~2 вопроса.

## 5. Заявления

### CLI

- ✅ **МОЖНО:** «Archcore держит стоимость за задачу плоской с ростом базы; `CLAUDE.md` preload
  растёт линейно. Дальше ~27 доков Archcore дешевле; при 320 доках — в 4.6 раза».
- ✅ **МОЖНО:** «Archcore на паритете с "файлы + греп" по стоимости за задачу и не требует
  обслуживания индекса».
- ✅ **МОЖНО:** «Archcore: 100% качество на всех размерах базы и условиях».
- ❌ **НЕЛЬЗЯ:** «Archcore экономит токены против любого бейзлайна» — не выигрывает у грепа.
- ❌ **НЕЛЬЗЯ:** «Результаты обобщаются на реальные базы» — задачи с неугадываемыми токенами
  сложнее большинства реальных запросов.

### Plugin

- ✅ **МОЖНО:** «Батч-сессии из 4+ вопросов на область экономят 74% за вопрос против отдельных
  вызовов. Подтверждено по всем 5 доменам, 3 trial, 100% pass rate».
- ✅ **МОЖНО:** «Plugin не хуже или лучше сырого MCP при N=4. Для ADR-документов Plugin на 20%
  дешевле сырого MCP при батче 4».
- ✅ **МОЖНО:** «Plugin даёт в 8× более стабильную стоимость за вопрос».
- ❌ **НЕЛЬЗЯ:** «Plugin экономит токены на одиночных запросах» — на 29% дороже.
- ❌ **НЕЛЬЗЯ:** «Скилл контекста ускоряет поиск» — экономия сессионная, не поисковая.

## 6. Оговорки

1. **Синтетическая база с неугадываемыми токенами** — сложнее большинства реальных запросов.
2. **Одна модель (sonnet), одно репо (chi).** Только однофакторные задачи.
3. **Индекс B2 считается идеально поддерживаемым и бесплатным** — жёсткий бейзлайн.
4. **Plugin-батч: 5 доменов, 3 trial.** ADR-насыщенные базы могут показать бо́льшее преимущество.
5. **Realistic-метрика — реконструкция**; cold/warm-границы: `scale/results/FINDINGS_SCALE_raw.md`.
