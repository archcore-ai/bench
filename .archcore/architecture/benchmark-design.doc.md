## Overview

Two independent test rigs measure token cost and answer quality for the Archcore CLI/plugin relative to alternative knowledge-access strategies. The sanity rig proves the harness works on a trivial KB; the scale rig sweeps KB size to locate the crossover point where MCP retrieval becomes cheaper than context preloading; the plugin rig measures session amortization economics of the `--plugin-dir` integration.

## Rigs

| Rig | Location | KB size | Arms | Purpose |
|-----|----------|---------|------|---------|
| Sanity | `bench/` root | 1 doc | A / B / C | Harness validation |
| Scale — CLI | `bench/scale/` | 320 docs (5 domains × 64) | A / B1 / B2 / B3 / C | Cost crossover + workload measurement |
| Scale — Plugin | `bench/scale/` | 80 docs (N=80 fixed) | D / C-batch | Batch session economics |

## Isolation Mechanism

Every arm runs `claude -p` with `--strict-mcp-config --mcp-config <file>`. Without `--strict-mcp-config`, headless Claude pulls in the user's global MCP config (~40 K tokens of noise that would pollute token counts and break arm comparisons). Strict mode zeroes that out.

- Arm C uses `mcp-archcore.json`: activates the archcore MCP server pointing at the arm's working directory.
- All other CLI arms use `mcp-empty.json`: no MCP servers.
- Each (N, arm) pair gets its own directory copy, isolating `CLAUDE.md` and `.archcore/` presence.
- The user's default config dir (OAuth / macOS Keychain) is retained for auth.
- Residual: globally-installed plugin skill frontmatter is constant across arms → cancels in deltas.

**Plugin rig (D-arm) isolation:** `run_plugin.sh` passes `--plugin-dir /path/to/plugin` in addition to `--strict-mcp-config --mcp-config mcp-archcore.json`. The `/archcore:context <domain>` skill is invoked as the first line of each batch prompt. C-batch uses the same MCP config but no `--plugin-dir`.

## Six-Arm Design

All arms share the same chi source substrate and Read/Grep/Glob tools. B1/B2/B3/C carry identical documents — only the access method differs. A is the floor. D extends C with the Claude Code plugin integration.

| Arm | Knowledge | Index | Extra tools | Strategy |
|-----|-----------|-------|-------------|----------|
| A — cold | none | — | — | Floor; agent reads source only |
| B1 — preload | N docs concatenated into CLAUDE.md | none | — | Dump everything into context |
| B2 — index+grep | N docs as `.archcore/*.md` + CLAUDE.md index (paths + topics, no answer tokens) | hand-maintained | — | Docs-as-files with curated map |
| B3 — blind grep | Same N doc files, no index | none | — | Agent discovers via Grep |
| C — archcore | Same N doc files | none (auto) | archcore MCP | MCP retrieval |
| D — plugin | Same N doc files | none (auto) | archcore MCP + `--plugin-dir` | MCP + `/archcore:context` skill |

`build_arm.py` materializes each arm:
- **A**: chi source only.
- **B1**: chi source + CLAUDE.md = bodies of first N docs concatenated (frontmatter stripped).
- **B2**: chi source + `.archcore/` with N doc files + CLAUDE.md index (paths + topic, no answer tokens).
- **B3**: chi source + `.archcore/` with N doc files, no CLAUDE.md.
- **C**: chi source + `.archcore/` with N doc files, no CLAUDE.md (MCP supplies discovery + retrieval).
- **D**: same as C + `--plugin-dir` activating `/archcore:context` skill.

Why B2 and B3 both exist: B2 gives the "markdown in repo" approach a perfect free index — the hardest baseline. B3 removes the index; agent discovers via Grep. The fair duel is **C vs B3**: same files, same zero index-maintenance burden, MCP search vs. plain Grep. **D vs C-batch** isolates the marginal value of the Plugin skill over raw MCP in batch sessions.

## Three Phases

**Crossover phase** — one fixed anchor task; KB size swept N ∈ {1, 20, 80, 160, 320}; arms A–C; 5 trials per cell. Reveals how per-task cost depends on KB size and locates the crossover point.

**Workload phase** — fixed KB N = 80; 20 tasks spanning all 5 domains (one fact each); arms B2/B3/C only; 3 trials. Measures total cost of a realistic task suite.

**Plugin batch phase** — fixed KB N = 80; batch sizes N_q ∈ {1, 2, 4, 8}; arms D and C-batch; 3 trials × 5 domains × 4 batch sizes = 120 measurements. Measures per-question cost as session amortization benefit grows with batch size.

For the normative definitions of arm construction, honesty invariants, cost metric formula, grading rules, and isolation requirements see the methodology spec.

## When to Use Which

| Scenario | Recommended approach |
|---|---|
| Script or one-off lookup | CLI (separate `claude -p` per question) |
| Interactive session, 2–3 questions about one area | Either — Plugin break-even is ~N=2 |
| Interactive session, 4+ questions about one area | **Plugin** — 74% cheaper per question |
| KB growing beyond ~27 docs | CLI beats `CLAUDE.md` preload; cost stays flat |
| Replacing a large `CLAUDE.md` | CLI — 4.6× cheaper at 320 docs |
| Need consistent, predictable cost | Plugin — 12× tighter cost variance at N=4 |
